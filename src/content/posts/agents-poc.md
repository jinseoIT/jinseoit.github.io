🧔 : "야, 나 지인 중에 중장년층 대상으로 유튜브 하시는 분이 계신데, 앱을 하나 만들고 싶다고 하시거든."

👱🏻‍♂️ : "아 그래서?"

술자리 특유의 흐름이랄까. "그럼 우리 AI 에이전트로 POC 하나 빠르게 만들어드리면 되겠네" 하고 별생각 없이 받았다. 근데 말이 나온 김에 진짜로 해보기로 했다. Claude Code + MCP 조합으로, 빠르면 하루 안에 될 것 같았다.

## 에이전트 구조를 어떻게 짰냐면

처음에 "Claude한테 앱 다 만들어줘" 하면 스파게티 코드 범벅에 토큰만 낭비할게 뻔하기에(최근 Claude 토큰 먹는 양이 내부로직 변경이 있는듯?..) 프롬프트는 길어질수록 컨텍스트가 흔들리고, 앞에서 한 말을 뒤에서 잊어버린다. 한 번에 다 시키면 안 된다는 걸 날라간 토큰들의 유산으로 삼고 작업을 시작해보자.

먼저 역할을 쪼갰다.

```
Orchestration Agent
│
├─ 1. Benchmark Agent  → 경쟁 앱 UX 분석
├─ 2. Product Agent    → PRD, 유저 플로우
├─ 3. Design Agent     → Figma 디자인 (MCP)
├─ 4. Frontend Agent   → 코드 생성
└─ 5. QA Agent         → 테스트 코드
```

`agents/` 폴더에 각 에이전트 역할을 마크다운으로 써두고, Claude Code 세션 열면서 "이 파일 읽고 이 역할로 행동해줘" 하면 된다. 단순한데 꽤 잘 됐다.

한 가지 신경 쓴 건 게이팅 조건이었다. 예를 들어 Design Agent는 PRD 파일이 없으면 실행이 안 되게 했다. 에이전트가 "이전 산출물 없어요, Benchmark 먼저 돌리셔야 해요"라고 멈추는 게, 처음엔 귀찮았는데 나중엔 이게 맞다 싶었다. 의존 관계를 명확하게 강제해주니까.

## Benchmark Agent: 경쟁사 분석을 30분에

Tinder, Bumble, Hinge, 골드스푼, 이음, 정오의데이트 여섯 개 앱을 분석하게 했다. 온보딩 단계 수, 프로필 구조, 매칭 방식, 신뢰 장치.

결과가 `output/benchmark/analysis.md`에 쌓이는 걸 보면서 좀 멍했다. 내가 직접 했으면 각 앱 설치하고, 가입하고, 플로우 따라가면서 캡처하고, 정리하는 데 이틀은 걸렸을 텐데. 30분이었다.

여기서 나온 인사이트 중에 나중에 제일 많이 쓴 게 이거다.

- 스와이프 방식은 즉흥적 판단을 유도한다 → 중장년층 타겟에 안 맞음
- 신뢰 장치가 없어서 허위 프로필 불안이 이탈의 핵심 원인
- 첫 메시지 부담이 크다 → 대화 시작 지원이 필요함

## Product Agent: PRD가 한 시간에

벤치마크 결과를 그대로 입력으로 넘겼다. 컨텍스트 체이닝이라는 게 별게 아니고, 앞 에이전트의 output이 다음 에이전트의 input이 되는 것뿐이다. 근데 이게 생각보다 훨씬 효과적이었다.

페르소나도 나름 구체적으로 나왔다.

> **김민준, 48세, 중소기업 팀장.** 이혼 후 3년, 재혼 희망. 카카오톡은 잘 쓰지만 새 앱은 낯설어함.
>
> **이수연, 44세, 간호사.** 미혼, 결혼 희망. SNS는 하지만 데이팅 앱은 처음.

FE 관점에서 내가 가장 중요하게 본 기능은 두 가지였다.

**오늘의 인연** — 하루 최대 5명 AI 큐레이션. 스와이프 대신 신중한 선택을 유도하는 구조.

**아이스브레이킹** — 공통점 기반 대화 질문 자동 생성. 첫 메시지 부담이 이탈 핵심이라는 벤치마크 결과를 기능으로 연결한 거다. 이 연결고리가 자연스럽게 나온 게 인상적이었다.

## Design Agent + Figma MCP

MCP(Model Context Protocol)가 이 단계에서 진짜 역할을 했다. Claude가 외부 도구와 직접 통신할 수 있게 해주는 프로토콜인데, Figma MCP를 연결하면 Design Agent가 명세를 만드는 동시에 Figma에 프레임을 직접 생성한다.

에이전트한테 준 디자인 원칙은 간단했다.

- 본문 16px 이상, 제목 20px 이상
- 터치 영역 44px 이상
- 화면당 핵심 액션 1~2개
- 신뢰감 있는 색상 (파란 계열, 과한 원색은 배제)

디자이너 없이 Figma 파일이 생기는 걸 보면서 솔직히 "이게 된다고?" 했다. 완성도가 높진 않다. 그냥 레이아웃 방향 잡는 수준이다. 근데 POC 단계에서 그게 어디냐.

## Frontend Agent: 여기서 내가 직접 결정한 것들

코드는 에이전트가 생성하지만, 아키텍처 결정은 내가 한다.

```
React Native Expo (앱스토어 배포)
└── WebView
    └── Next.js 웹앱 (실제 UI/로직)
        └── Supabase (Auth, DB, Realtime)
```

이 구조를 선택한 건 의도가 있었다. 앱스토어 배포는 필요하지만 네이티브 코드 두 벌을 POC 단계에서 관리하는 건 과하다. Next.js에서 UI/로직 전부 처리하고, Expo는 WebView 껍데기만. 딥링크나 푸시 알림처럼 진짜 네이티브 기능이 필요할 때만 postMessage로 브릿지한다.

Supabase를 고른 이유는 세 가지였다. SMS OTP Auth가 기본 제공되고, Realtime으로 채팅 구현이 빠르고, `supabase gen types`로 타입 자동 생성이 된다.

### 버그 잡다가 퍼널 패턴으로 갈아엎음

회원가입 플로우에서 에이전트가 계속 버그를 냈다. 스텝 이동 로직이 꼬이고, 특정 조건에서 스텝이 건너뛰어지거나 뒤로 돌아가지 않는 문제였다. 프롬프트를 고쳐서 재생성시켜도 비슷한 위치에서 비슷한 버그가 나왔다.

코드를 보니까 이유가 있었다.

```typescript
// 숫자 인덱스 기반
const [step, setStep] = useState(0);

{step === 0 && <PhoneStep />}
{step === 1 && <BasicInfoStep />}
{step === 2 && <PhotoStep />}
```

`step === 2`가 뭔지 코드만 봐서는 모른다. 에이전트도 마찬가지였을 거다. 숫자 인덱스로 관리하니까 스텝이 추가될 때마다 어디서 무슨 숫자를 쓰는지 추적이 안 됐고, 그게 버그로 터졌다. 구조 자체가 에러를 유도하고 있었던 거다.

어차피 손댈 거면 제대로 바꾸자 싶어서 퍼널 패턴으로 리팩토링했다.

```typescript
const STEPS = ['phone', 'basicInfo', 'photo', 'bio', 'trust'] as const;

const { Funnel, setStep } = useFunnel(STEPS, 'phone');

return (
  <Funnel>
    <Funnel.Step name="phone">
      <PhoneStep onVerified={() => setStep('basicInfo')} />
    </Funnel.Step>
    <Funnel.Step name="basicInfo">
      <BasicInfoStep onNext={() => setStep('photo')} />
    </Funnel.Step>
    {/* ... */}
  </Funnel>
);
```

차이가 꽤 크다.

- **이름으로 이동한다.** `setStep('basicInfo')`라고 쓰면 어디로 가는지 코드에서 바로 보인다. 숫자 인덱스처럼 따라가지 않아도 된다.
- **각 스텝이 독립적이다.** `<Funnel.Step name="photo">`는 그 스텝 안에서만 책임진다. 다른 스텝 건드리지 않고 하나만 수정할 수 있다.
- **순서 변경이 선언적이다.** 스텝 추가하거나 순서 바꿀 때 `STEPS` 배열 하나만 손대면 된다. 조건부 렌더링 전체를 뒤질 필요가 없다.

`useFunnel` 훅 자체는 30줄 남짓이다. 에이전트한테 다시 붙여줬더니 이후로 스텝 이동 관련 버그가 사라졌다. 구조가 명확하니까 에이전트도 덜 헷갈리는 것 같았다.

## 하루 끝에 남은 것들

- 경쟁사 UX 분석 리포트 (6개 앱)

스택: Next.js 15 + TypeScript + Tailwind + Supabase + React Native Expo

## 안 된 것도 있었다

**Supabase 타입 연동이 생각보다 까다로웠다.** `supabase gen types`로 생성한 타입이랑 실제 코드 사이에서 `never` 타입 에러가 계속 났다. 에이전트가 스키마를 100% 반영 못 한 부분이었고, 이건 직접 잡아야 했다.

**pnpm 환경 이슈.** `@types/react` 심링크가 제대로 안 생기는 건지 새 파일 만들 때마다 VS Code TypeScript 서버가 tsconfig를 못 인식하는 경우가 생겼다. `pnpm install` 한 번 돌리고 TS 서버 재시작하면 됐는데, 이런 환경 이슈는 에이전트가 못 잡는다.

**디자인 완성도는 한계가 있다.** Figma MCP 결과물은 방향을 잡는 용도다. 실제 프로덕트 디자인은 여전히 디자이너가 필요하다는 걸 다시 확인했다.

## 결국 뭘 얻었냐면

이 실험에서 가장 크게 느낀 건 컨텍스트 체이닝의 힘이었다. 벤치마크에서 나온 "첫 메시지 부담" 인사이트가 PRD 기능으로, 그게 실제 UI로 자연스럽게 연결되는 흐름. 에이전트가 앞 단계 산출물을 들고 다음 단계로 넘어가니까 일관성이 유지됐다.

대체할 수 없는 부분도 분명하다. 아키텍처 결정, 기술 트레이드오프, 코드 품질 검수. 에이전트가 만든 코드를 검토하고 방향을 잡는 건 여전히 사람 몫이다.

AI 에이전트가 개발 속도를 올려주는 도구인 건 맞다. 근데 판단을 대신해주는 건 아니다. Orchestration Agent가 파이프라인 돌리는 동안 나는 아키텍처 설계하고, 기술 결정 내리고, 산출물 검증했다. 반복 작업이 빠져나간 자리에 판단할 게 더 많이 남았다. 🧐
